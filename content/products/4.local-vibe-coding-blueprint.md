## Who This Is For

You want an AI coding assistant that understands your codebase, keeps your source code private, and doesn't bill you per token. This blueprint is the practical layer underneath the [Local Vibe Coding series](/categories/local-vibe-coding): the hardware decisions, the manifests, and the router-and-skills code that turn "I have a GPU" into "I have a working local coding assistant" — one with a real job split between a local Principal Developer and Claude Code as the external Principal Architect, not a single model doing everything.

## Why It's Different From Just Reading the Blog

The blog series explains the reasoning, the trade-offs, and the honest failure modes. This blueprint is the working starting point: the hardware split across the RTX 3090 (the coding agent), the Jetson fleet (retrieval, background workers, utility models), and the storage node (Weaviate, Postgres, MinIO); the Kubernetes manifests that schedule all three; the repository indexing scripts; the Vibe Fleet VS Code extension that puts the whole fleet behind your editor's command palette; and the OpenClaw router with its full skill set — `architecture-review`, `repository-memory`, `coding-agent`, `code-review`, `security-review`, `test-runner`, `deployment`, `ask-claude-fix` — you'd otherwise have to reverse-engineer from a dozen posts. It also includes the `agent_tasks` schema that turns every local miss into LoRA training data instead of a discarded failure.

Nothing here was written for the blueprint after the fact — every manifest, skill folder, and script is copied from the cluster the series is actually built on, config values redacted.

## The VS Code Extension: Vibe Fleet

The newest piece of the blueprint puts the whole system inside your editor. Vibe Fleet is a VS Code extension (shipped as both an installable VSIX and full TypeScript source, zero runtime dependencies) that treats every GPU you own as a **worker with roles**:

| Role | What that hardware does |
|---|---|
| `draft` | writes code in the draft → verify loop |
| `verify` | reviews changesets and your own diffs, with a confidence score |
| `embed` | powers the semantic code index (a Jetson handles this fine) |
| `chat` | explain-this-code, ask-the-fleet Q&A |
| `utility` | commit messages, handoff compression |

A priority router picks the best healthy worker per role, so a slow-but-careful drafter and a fast fallback coexist. Workers and databases are plain `settings.json` entries — any OpenAI-compatible or Ollama endpoint works — and the sidebar shows live health dots for every box and every database (Weaviate, Redis, Postgres, OpenSearch, ClickHouse, MongoDB, MinIO) so you know at a glance what's up before you hand work to it.

## Setup Workflow

1. **Install**: `code --install-extension vibe-fleet-0.1.0.vsix`, reload VS Code.
2. **Register your hardware**: load the included home-lab preset and edit the IPs to yours, or add each box through the guided `Add Worker` wizard. One `ollama pull nomic-embed-text` on your embedding node (274 MB — runs comfortably on a Jetson Orin Nano).
3. **Index the workspace** (`Vibe Fleet: Index Workspace`): your code is chunked, embedded on your own hardware, and stored in Weaviate. Incremental after the first run — unchanged files are hash-skipped.
4. **Learn your style** (`Vibe Fleet: Learn Code Style`): a local model samples the repo and writes a compact style guide that rides along with every future draft, so generated code matches *your* conventions instead of generic tutorial style.

## Daily Workflow

The free tier of your day — none of this touches a paid API:

- **Semantic code search** in plain English, jumping straight to file and line
- **Explain Selection** from the right-click menu
- **Generate Commit Message** — lands directly in the Source Control input box
- **Review My Diff** — your verifier model reads `git diff HEAD` like a strict senior reviewer before you commit
- **Run Task with Local Agents** — describe a change; the drafter proposes a complete changeset (models never touch disk directly), the verifier approves or rejects with specific issues, and approved changes are committed on a fresh `vibe/<slug>` git branch. Everything stays one `git revert` from undone.

## Escalation Without the Token Bill

Local models are free but not always enough — and the worst failure mode is a local agent looping forever on a task above its weight class. The loop escalates to Claude Code when the verifier rejects a configurable number of rounds in a row, or approves with self-reported confidence below your threshold (default 0.6).

Escalation is where the **token saver** earns its place: before Claude sees anything, proposed changes are rewritten as unified diffs instead of full file dumps, the attempt history is deduplicated down to the task plus the distinct verifier complaints, and if the handoff still exceeds your token budget (default ~6k), a free local model compresses it further. The notification tells you exactly how many tokens the saver trimmed — **and how many tokens of free local work the fleet already did** (every draft, revision round, review pass, and embedding is work Claude never has to repeat). A session ledger keeps the running total one hover away in the status bar. You escalate with a dense, cheap brief — not a transcript.

Every knob — rounds before escalation, confidence threshold, token budget, ask/auto/never — is a setting, so the same extension serves a cautious beginner and a fully autonomous overnight loop.
