## Engineering Overview

Business Benefit Alliance is a full-stack group insurance enrollment platform built to manage the complete lifecycle of small-business benefit enrollment — from company registration and employee onboarding through signed application submission, payment authorization, and insurance card delivery.

The core engineering challenge is not the form itself — it is coordinating a multi-role, multi-step workflow that produces legally traceable PDFs, captures and embeds digital signatures, writes documents to S3, and exposes the right data surface to each of three distinct user roles. Every step in the enrollment flow has a corresponding audit entry, and every signed document is generated server-side and stored durably before the user is advanced to the next step.

The Nitro server enforces the enrollment state machine: no employee can skip steps, no business owner can proceed to payment authorization before all required applications are submitted, and the app admin cannot fulfill enrollment until the owner has completed payment authorization. Role boundaries are enforced at the service layer, not just the route level.

## Technical Highlights

::HighlightGrid

::HighlightCard{icon="mdi:email-fast" title="Invite-Based Employee Onboarding"}
The business owner sends invite emails from the dashboard. Each invite carries a unique business code. When an employee signs up using that code, the system automatically assigns them to the correct company account — no manual admin step required. Invalid or expired codes are rejected at the API layer before account creation proceeds.
::

::HighlightCard{icon="mdi:file-sign" title="Digital Signature Capture → PDF Embedding"}
Each insurance application form includes a signature pad. On submission, the captured signature is passed server-side where it is composited directly into the generated PDF alongside the applicant's form data. The resulting document is the legally signed application — not a separate attachment — and is stored in S3 before the step is marked complete.
::

::HighlightCard{icon="mdi:shield-account" title="Multi-Step Insurance Application"}
Owners and employees complete a structured enrollment flow: personal details, dependent/family information, insurance plan selection, and dental and vision opt-in. Each section is validated server-side before the next is unlocked. Partial saves are persisted so applicants can return without losing progress.
::

::HighlightCard{icon="mdi:file-document-multiple" title="Consolidated Payment Authorization PDF"}
After all employee applications are submitted, the business owner completes a payment authorization form. The system generates a consolidated PDF containing the owner's authorization, a summary of all employee plan selections, and every captured signature — producing a single document that covers the full enrollment batch.
::

::HighlightCard{icon="mdi:history" title="Audit Trail Per Enrollment"}
Every meaningful action in the enrollment workflow — application started, step completed, signature captured, PDF generated, document uploaded — is written to an append-only audit log tied to the company's enrollment record. The audit trail is accessible to the app admin and is included in the S3-stored document package.
::

::HighlightCard{icon="mdi:card-account-details" title="Admin Fulfillment & Insurance Card Delivery"}
Once the app admin reviews and fulfills an enrollment, they upload the insurance cards directly from the admin dashboard. Cards are stored in S3 and immediately available for download by the business owner and each enrolled employee from their respective dashboards — no email attachments, no manual forwarding.
::

## System Architecture

Business Benefit Alliance follows a role-driven architecture with three dashboard surfaces sharing a single Nitro API layer. Each role — business owner, employee, and app admin — has a scoped data access profile enforced at the service layer. The enrollment workflow is modelled as an explicit state machine: each company record has an enrollment status that gates what actions are available at any point in time.

## Challenges Solved

::ChallengeGrid

::ChallengeCard{title="Business Code-Based Employee Assignment"}

#problem
Employees needed to be automatically linked to the correct company on sign-up without requiring manual admin intervention or exposing all company records for selection.

#solution
Each company gets a unique business code generated on registration. Invite emails carry that code. On sign-up, the submitted code is validated against the company table — if valid, the new employee record is linked to that company immediately. Invalid, expired, or already-used codes are rejected before account creation proceeds, so partial registrations never leave orphaned records.

::

::ChallengeCard{title="Embedding Digital Signatures into Generated PDFs"}

#problem
The signature capture happened client-side but the PDF had to be a single coherent document — not a base document with a separately attached signature image that could be detached or disputed.

#solution
The signature is captured as vector data client-side and transmitted to the server on form submission. The PDF generation pipeline renders the complete form layout, then composites the signature directly into the document at the designated field coordinates before the file is written to S3. What lands in storage is one fully composited PDF — the signature is part of the document, not appended to it.

::

::ChallengeCard{title="Maintaining a Tamper-Evident Audit Trail"}

#problem
Insurance enrollment requires a clear record of who did what and when. A simple updated_at timestamp on form records was insufficient — the system needed an immutable history of every meaningful workflow event.

#solution
A dedicated audit log table stores append-only entries for every enrollment event: form opened, step completed, signature captured, PDF generated, document written to S3, enrollment fulfilled. Each entry records the actor, timestamp, action type, and a reference to the affected record. The log is never updated or deleted — only appended to — and the S3 document package includes an export of the audit entries for that enrollment.

::

::ChallengeCard{title="Soft Delete Without Corrupting Enrollment Records"}

#problem
Business owners needed to remove employees from active enrollment without destroying the historical application data those employees had already submitted — especially if an employee's partial application was part of a completed enrollment batch.

#solution
Employee records use a soft delete flag rather than a hard delete. Soft-deleted employees are excluded from active counts, invite flows, and dashboard lists. Their application records remain intact and associated with the enrollment batch. A restore action flips the flag back — no data is lost, and the employee's application history is immediately available again. Hard deletes are not exposed in the UI.

::

::ChallengeCard{title="Role-Scoped Data Access Across Three Dashboards"}

#problem
Three distinct user roles — business owner, employee, and app admin — needed to see different subsets of the same underlying data. Route guards alone were insufficient; a determined user could call API endpoints directly and receive data outside their role.

#solution
Role enforcement is applied at the Nitro service layer, not just at route guards. Every API handler resolves the authenticated user's role before constructing the database query. Employees only receive their own records. Business owners receive their company's records. The app admin receives cross-company data. No handler returns a superset of what the role is authorized to see — UI route guards are a secondary UX convenience, not the security boundary.

::

::ChallengeCard{title="Sequential Workflow Enforcement Across Multiple Users"}

#problem
The enrollment workflow involves multiple actors — each employee submits their own application, then the owner submits payment authorization — and the system needed to prevent any actor from advancing out of sequence, even with concurrent requests.

#solution
The enrollment record on the company carries an explicit status field modelled as a state machine. Each API action that advances the workflow checks the current status before writing. Transitions are atomic database operations — the status update and the related record write happen in the same transaction. If an employee tries to submit an application before the owner has opened enrollment, or if the owner tries to submit payment authorization before all employee applications are complete, the request is rejected with a specific error rather than silently accepted.

::

## Engineering Perspective

::HighlightGrid

::HighlightCard{icon="logos:nuxt-icon" title="Frontend — Nuxt 3"}
Vue 3 Composition API throughout. SPA mode for authenticated dashboard routes with Pinia stores scoped per role. Skeleton loaders implemented on all three dashboards to handle async data fetch without layout shift. Signature pad integrated as a client-side component with output normalized before transmission to the server. Strict TypeScript shared across client and Nitro layer.
::

::HighlightCard{icon="logos:nodejs-icon" title="API Layer — Nitro Server"}
H3 event handlers with Zod schema validation at every enrollment endpoint. Enrollment state machine logic lives in the service layer — controllers are thin and do not own workflow logic. Role resolution happens in middleware before the handler runs. PDF generation and S3 writes are awaited synchronously within the submission handler so the client receives confirmation only after the document is durably stored.
::

::HighlightCard{icon="mdi:file-pdf-box" title="Document Generation Pipeline"}
PDFs are generated server-side on form submission using a structured template per document type. Signature data composited at defined coordinates. Generated documents are streamed to AWS S3 via the SDK; the S3 object key is written to the database in the same transaction as the enrollment step update. Pre-signed S3 URLs are generated at request time with short TTLs — documents are never served from a public bucket.
::

::HighlightCard{icon="mdi:email-fast" title="Invite & Onboarding Flow"}
Business codes are generated as short, collision-resistant strings scoped to the company record. Invite dispatch uses Nodemailer with a template carrying the code and a direct sign-up link. Code validation is a single indexed lookup — no joins required. Codes are marked used on successful registration; re-submission with the same code is rejected before any write occurs.
::

::HighlightCard{icon="mdi:shield-lock" title="Auth & Role Enforcement"}
JWT-based auth with role claim embedded in the token. Role is verified server-side against the database on protected routes — the token claim alone is not trusted for data access decisions. Soft delete state is checked on authentication so a deleted employee cannot log in until restored. Session invalidation is handled at the token layer without requiring a Redis dependency.
::

::HighlightCard{icon="mdi:code-braces-box" title="Type Safety — End to End"}
Strict TypeScript across the Nuxt frontend and Nitro server. Zod schemas at API boundaries generate both runtime validation and inferred TypeScript types. Prisma client types flow from schema migrations. Enrollment status values are modelled as a TypeScript enum shared across client and server — invalid transitions are caught at compile time before they reach the database.
::

## Database Design

::HighlightGrid

::HighlightCard{icon="logos:postgresql" title="PostgreSQL Schema"}
Company and employee records are related by a foreign key to the company table; the business code is a unique indexed column on the company record. Enrollment applications store step-level progress with a JSONB column for flexible form data per step. A separate audit log table is append-only with no update or delete paths exposed via the ORM. Insurance card records are linked to both the company and individual employee with S3 key columns and generated-at timestamps.
::

::HighlightCard{icon="mdi:file-document-outline" title="Document Record Design"}
Each generated PDF gets a document record storing the S3 key, document type (application | payment-auth | insurance-card), the actor who triggered generation, and the enrollment batch it belongs to. Pre-signed URL generation is derived from the key at request time — the URL is never stored. Document records are immutable after creation; a new record is created if a document is regenerated rather than overwriting the existing row.
::

::HighlightCard{icon="mdi:state-machine" title="Enrollment State Machine"}
The company enrollment record carries an explicit status column with a defined set of valid transitions: draft → open → applications-complete → payment-submitted → fulfilled. The Prisma service layer enforces valid transitions using a check before every status-advancing write. Invalid transitions return a typed error. This makes the enrollment lifecycle queryable and auditable without reconstructing it from event history.
::

::HighlightCard{icon="mdi:database-sync" title="Prisma Migration Strategy"}
All schema changes go through Prisma migrate — no hand-written SQL in production. Soft delete is implemented as a `deletedAt` nullable timestamp column; all queries that should exclude deleted records use a shared Prisma middleware filter rather than per-query `where` conditions. Seeder scripts populate plan options and test company accounts for local development.
::
