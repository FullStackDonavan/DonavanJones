## Engineering Overview

Kingdom Tools is a Kubernetes-native internal developer platform that extends the cluster with a custom resource definition (`KingdomService`) and a continuous reconciliation controller. It's not a dashboard layered on top of kubectl — it's a platform extension that defines how services are created, what infrastructure they own, and ensures declared state matches actual state.

The core workflow: declare a service as a KingdomService CR, `kubectl apply` it, and the controller provisions isolated infrastructure across nine data layers within seconds. If something drifts, the controller fixes it. If a resource is orphaned, the audit engine flags it.

```yaml
apiVersion: kingdom.donavanjones.com/v1alpha1
kind: KingdomService
metadata:
  name: my-service
spec:
  template: ai-agent
```

```bash
$ kubectl get ks
NAME         TEMPLATE    PHASE   AGE
my-service   ai-agent    Ready   12s
```

## Technical Highlights

::HighlightGrid

::HighlightCard{icon="mdi:kubernetes" title="KingdomService CRD & Controller"}
Services are defined as Kubernetes custom resources with a structured spec (template, resources, deployment config) and a status subresource (phase, per-resource conditions, observedGeneration). A background controller thread watches all CRs via the Kubernetes watch API and continuously reconciles desired state to actual infrastructure across nine data layers.
::

::HighlightCard{icon="mdi:rocket-launch" title="Template-Based Provisioning"}
Ten service templates cover every workload type: web apps, AI agents, scrapers, API services, background workers, document stores, search services, event-driven services, workflow engines, and analytics pipelines. Apply a CR with a template name and the controller creates isolated resources with no manual steps.
::

::HighlightCard{icon="mdi:shield-search" title="Audit & Drift Detection"}
The audit engine cross-references every database, bucket, Redis slot, Weaviate collection, MongoDB database, OpenSearch index, Kafka topic, Temporal namespace, and ClickHouse database against the service catalog. Orphans and missing resources are flagged. The controller automatically fixes drift on the next reconcile cycle.
::

::HighlightCard{icon="mdi:sync" title="Continuous Reconciliation"}
The controller runs as a daemon thread inside the FastAPI process. It watches for CR changes, handles watch expiry with automatic relist, uses exponential backoff on errors, and tracks observedGeneration to skip already-reconciled services. Finalizers handle clean deletion.
::

::HighlightCard{icon="mdi:key-variant" title="Secrets Management"}
Per-service K8s Secrets (SECRET_KEY, JWT_SECRET, API_KEY) generated with cryptographically secure values. Add custom keys, rotate individual keys or all at once. Secrets are labeled `managed-by: kingdom-tools` and never displayed in the UI.
::

::HighlightCard{icon="mdi:chart-line" title="Grafana Dashboard Provisioning"}
One-click per-service Grafana dashboards with seven panels: request rate, error rate, p95 latency, memory usage, CPU usage, pod restarts, and a Loki log viewer. All panels scoped to the service's namespace and pod pattern.

## System Architecture

Kingdom Tools runs as a single FastAPI pod in the `kingdom-tools` namespace, exposed via NodePort 30800. The reconciliation controller runs as a background thread in the same process, watching `KingdomService` CRs via the Kubernetes watch API.

The pod uses an in-cluster ServiceAccount with a ClusterRole scoped to read pods/services/nodes, patch deployments, manage secrets, and full CRUD on the `kingdomservices` custom resource and its status subresource.

The backend connects to nine data layers (PostgreSQL, MongoDB, MinIO, Redis, Weaviate, OpenSearch, Kafka, Temporal, ClickHouse) plus Grafana, Prometheus, and Loki for observability, and Gitea for container registry integration. The frontend is vanilla JS and CSS with no build step.

## Challenges Solved

::ChallengeGrid

::ChallengeCard{title="Imperative Infrastructure Management"}

#problem
Creating a new service meant manually provisioning resources across multiple systems, assembling connection strings, and writing deployment YAML — with no record of what belongs to what and no way to detect when things drift.

#solution
The KingdomService CRD makes infrastructure declarative. One YAML file defines what a service needs. The controller provisions everything, updates status with per-resource conditions, and continuously reconciles. `kubectl get ks` shows the health of every service.

::

::ChallengeCard{title="Orphaned Resources After Service Removal"}

#problem
Deleting a deployment didn't clean up its database, bucket, search index, event topic, or vector collection. Over time, orphaned resources accumulated across nine data layers with no way to trace them back to their original service.

#solution
The audit engine scans all nine data layers and cross-references against the service catalog. Resources not claimed by any registered service are flagged as orphans. The reconciler can fix missing resources, and finalizers on the CRD handle deletion cleanup.

::

::ChallengeCard{title="Secret Sprawl and Rotation"}

#problem
Each service needed manually generated secrets hardcoded into deployment manifests. No rotation, no central management, no way to audit which services had which keys.

#solution
Kingdom Tools generates cryptographically secure K8s Secrets per service, labeled for ownership tracking. Individual keys can be rotated from the dashboard without redeploying. The secrets are stored natively in Kubernetes, not in application config.

::

::ChallengeCard{title="No Per-Service Observability"}

#problem
Grafana dashboards had to be manually created for each service, with Prometheus queries hand-written to match the right namespace and pod patterns.

#solution
One-click dashboard creation generates seven-panel Grafana dashboards scoped to the service's namespace and pod labels. Request rate, error rate, latency, resource usage, and logs — all wired up automatically and tagged for easy filtering.
